# Terminal.Gui.Stanza: MVP Implementation Plan

**Date**: May 30, 2026  
**Goal**: Build an end-to-end MVP demonstrating the full Stanza paradigm—declarative views with attribute-driven layout, data binding, and generated `InitializeComponent()` code.

---

## Executive Summary

This plan outlines a minimal but complete implementation path to validate the Stanza architecture. The MVP covers:

- **IR Models**: Immutable records representing view declarations, bindings, and layout constraints
- **Runtime Foundation**: `BindableView<T>` base class, completed binding extensions, layout DSL
- **Source Generator**: Roslyn `IIncrementalGenerator` that parses decorated classes, resolves layout dependencies, and emits ordered initialization code
- **Validation**: End-to-end test demonstrating a working view; generator snapshot tests verifying emitted code

**Excluded from MVP**: Style system (`[TuiStyle]`), advanced layout math, DataTemplate binding, command parameters.

## Implementation Phases

### Phase 1: IR Models (Foundation)

**Location**: `src/Terminal.Gui.Stanza.Abstractions/`

Create immutable record types representing the Abstract Syntax Tree of a decorated view. These decouple Roslyn symbols from business logic (per ADR 0001).

**Files to create**:

1. **ViewDeclaration.cs**
   - Properties: `ClassName`, `Namespace`, `BaseType` (e.g., `BindableView<MyViewModel>`)
   - Represents a single `[TuiView]`-decorated class

2. **PropertyAssignment.cs**
   - Properties: `PropertyName`, `ValueExpression` (string), `IsLayoutConstraint` (bool)
   - Represents a property assignment (e.g., `Width = 20` or `Y = Pos.Bottom(other)`)

3. **BindingInfo.cs**
   - Properties: `PropertyName`, `ViewModelProperty`, `BindingMode` (enum: OneWay, TwoWay), `Converter` (optional)
   - Represents a binding created via extension (e.g., `BindText = nameof(vm.Name)`)

4. **LayoutConstraint.cs**
   - Properties: `SourceView`, `TargetProperty` (X or Y), `ConstraintType` (enum: Bottom, Right, etc.), `ReferencedView`
   - Represents a layout dependency (e.g., `viewB.Y = Pos.Bottom(viewA)` creates edge viewA → viewB)

5. **BindingMode.cs**

   ```csharp
   public enum BindingMode
   {
       OneWay,
       TwoWay
   }
   ```

```csharp
using Terminal.Gui;

/// <summary>
/// Base class for declarative Stanza views. Provides binding context and enforces generated code pattern.
/// </summary>
public abstract class BindableView<TViewModel> : View
{
    protected BindingContext BindingContext { get; private set; } = new();

    public TViewModel? ViewModel { get; set; }

    /// <summary>
    /// Called during initialization to wire up properties, layout, and bindings.
    /// This method is generated by TuiViewGenerator and should not be overridden.
    /// </summary>
    protected abstract void InitializeComponent();

    public BindableView()
    {
        InitializeComponent();
    }

    public override void Dispose()
    {
        BindingContext?.Dispose();
        base.Dispose();
    }
}
```

### Phase 4: Source Generator (Core Logic)

**Location**: `src/Terminal.Gui.Stanza.Generators/`

Replace `Class1.cs` with `TuiViewGenerator.cs`.

#### 4.1 High-Level Architecture

The generator consists of three stages:

1. **Parser**: Extract `[TuiView]`-decorated classes using Roslyn → construct IR models
2. **Resolver**: Analyze layout constraints → topologically sort views by dependency
3. **Emitter**: Generate C# `InitializeComponent()` method

#### 4.2 Parser Implementation

```csharp

internal class TuiViewParser

    public TuiViewParser(INamedTypeSymbol tuiViewAttributeSymbol)
    {

    /// Parse a ClassDeclarationSyntax decorated with [TuiView] into a ViewDeclaration IR.
```

#### 4.3 Dependency Resolver

```csharp
internal class DependencyResolver
{
    /// <summary>
    /// Topologically sort views by layout dependencies.
    /// If viewB.Y = Pos.Bottom(viewA), then viewA must be instantiated before viewB.
    /// </summary>
    public List<string> ResolveOrder(IEnumerable<LayoutConstraint> constraints, IEnumerable<string> allViewNames)
    {
        // Build adjacency list: viewName → List of views that depend on it
        var graph = new Dictionary<string, List<string>>();
        var inDegree = new Dictionary<string, int>();

        foreach (var view in allViewNames)
        {
            graph[view] = new List<string>();
            inDegree[view] = 0;
        }

        foreach (var constraint in constraints)
        {
            // Constraint: targetView depends on referencedView
            if (graph.ContainsKey(constraint.ReferencedView) && graph.ContainsKey(constraint.SourceView))
            {
                graph[constraint.ReferencedView].Add(constraint.SourceView);
                inDegree[constraint.SourceView]++;
            }
        }

        // Kahn's algorithm (topological sort)
        var queue = new Queue<string>();
        foreach (var view in allViewNames)
        {
            if (inDegree[view] == 0)
                queue.Enqueue(view);
        }

        var result = new List<string>();
        while (queue.Count > 0)
        {
            var view = queue.Dequeue();
            result.Add(view);

            foreach (var dependent in graph[view])
            {
                inDegree[dependent]--;
                if (inDegree[dependent] == 0)
                    queue.Enqueue(dependent);
            }
        }

        if (result.Count != allViewNames.Count())
        {
            // Circular dependency detected
            // TODO: Emit diagnostic error
        }

        return result;
    }
}
```

#### 4.4 Code Emitter

```csharp
internal class InitializeComponentEmitter
{
    /// <summary>
    /// Emit the InitializeComponent() method body.
    /// </summary>
    public string Emit(ViewDeclaration viewDecl, List<string> instantiationOrder)
    {
        var sb = new StringBuilder();
        sb.AppendLine("protected override void InitializeComponent()");
        sb.AppendLine("{");

        // 1. Instantiate views in dependency order
        sb.AppendLine("    // Instantiate views");
        foreach (var viewName in instantiationOrder)
        {
            // TODO: Extract the view's type from viewDecl.Assignments
            sb.AppendLine($"    var {viewName} = new Label(); // TODO: infer type");
        }

        sb.AppendLine();

        // 2. Apply layout constraints
        sb.AppendLine("    // Apply layout constraints");
        foreach (var constraint in viewDecl.LayoutConstraints)
        {
            var posMethod = constraint.ConstraintType == "Bottom" ? "Bottom" : "Right";
            sb.AppendLine($"    {constraint.SourceView}.{constraint.TargetProperty} = Pos.{posMethod}({constraint.ReferencedView});");
        }

        sb.AppendLine();

        // 3. Wire bindings
        sb.AppendLine("    // Wire bindings");
        foreach (var binding in viewDecl.Bindings)
        {
            var mode = binding.BindingMode == BindingMode.OneWay ? "BindingMode.OneWay" : "BindingMode.TwoWay";
            sb.AppendLine($"    BindingContext.AddBinding({binding.PropertyName}.Bind(vm => vm.{binding.ViewModelProperty}, {mode}));");
        }

        sb.AppendLine("}");

        return sb.ToString();
    }
}
```

## Decisions

1. **IR as Immutable Records**: Decouples Roslyn symbols from business logic. Easier to test in isolation.
2. **Topological Sort**: Deterministic, O(n) ordering of views. Circular dependencies caught at compile time.
3. **BindableView<T> as Abstract Base**: Enforces that generated code fills `InitializeComponent()`.
4. **No Runtime Reflection**: Generator emits direct method calls; zero runtime overhead.
5. **Kahn's Algorithm for DAG Sort**: Standard algorithm; well-tested and easy to understand.

## Out of Scope (MVP)

- **[TuiStyle] Attribute**: Style bundling and compile-time "splatting" (Phase 2 feature)
- **Advanced Layout**: Percentage sizing, anchor enums, margins (Phase 2)
- **DataTemplate Binding**: Itemized views, ObservableCollection (Phase 3)
- **Command Parameters**: Advanced MVVM patterns (Phase 3)
- **Two-Way Binding Variants**: Custom converters, validation (Phase 2)
- **Circular Dependency Recovery**: For now, errors; later, runtime resolution

## Further Considerations

### Circular Layout Dependencies

**Issue**: If layout refs form a cycle (A → B → A), topological sort fails.

**Decision**: Emit diagnostic error; require user to refactor.

**Future**: Could add "anchor" view concept (one view breaks the cycle).

### Extension Property Syntax

**Issue**: C# 14 extension properties are new; some tooling may lag.

**Decision**: Start with extensions only for MVP.

**Future**: Add fluent builder API as fallback if needed.

### ViewModel Change Notification

**Issue**: MVP test uses plain property with manual setter; doesn't observe changes.

**Decision**: For MVP, bindings are one-time at initialization.

**Future**: Support `INotifyPropertyChanged` and continuous observation.
